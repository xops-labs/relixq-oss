## [0.3.0] - 2026-06-11

### Added
- **Downloadable Community releases.** Pushing a `vX.Y.Z` tag now runs a release workflow (`.github/workflows/release.yml`) that tests, then publishes: per-platform archives (Windows x64 `.zip`; Linux x64 / macOS x64 / macOS arm64 `.tar.gz`) bundling `relixq` + `relixq-scan-code` + the community rules via GoReleaser (`.goreleaser.yaml`); `.deb`/`.rpm` packages; a SHA256 checksums file; a Windows `.msi` installer (WiX, `packaging/windows/`) that installs to Program Files and adds `relixq` to `PATH`, Authenticode-signed when signing secrets are configured; and the GHCR scanner image tagged `X.Y.Z` / `X.Y` / `sha-<short>` / `latest`, signed with cosign (keyless). Release docs in `docs/RELEASE.md`; README now leads with the download-and-scan evaluation path.
- **Zero-config CLI install layout.** `relixq` now resolves the scanner engine in order: explicit `RELIXQ_SCANNER_BIN` ŌåÆ `relixq-scan-code` next to the `relixq` executable ŌåÆ `PATH`; and when no `--rules` / `rule_dir` / `RELIXQ_RULE_DIR` is set it falls back to the bundled rules shipped with a release install (`rules/` or `rules-community/` beside the executable, or `../share/relixq/rules` for deb/rpm). An extracted release archive therefore scans with one command and no environment setup. `relixq doctor` checks the same resolution paths (and no longer probes the misnamed `relixq-scanner` binary), and `relixq-scan-code -version` prints the release-stamped version.
- **GitHub Action pins released images.** `github-action/action.yml` now pins the GHCR scanner image to a released version tag instead of `latest`; release prep bumps the pin so `github-action@vX.Y.Z` runs scanner image `X.Y.Z`. Main-branch image builds are tagged `main` + `sha-<short>` (`latest` now always means the latest release), and `Dockerfile.scanner` stamps the CLI version/commit via build args.
- **About page (`/about`).** A public, SEO-tagged marketing page (metadata + OpenGraph) explaining what Relix-Q OSS is, why post-quantum readiness matters (harvest-now-decrypt-later, FIPS 203/204/205), the feature set, the 31-language + config/cert coverage, the three-tier vulnerability taxonomy it finds, end-to-end security checks, and community/GitHub links. Linked from the header for signed-in and anonymous visitors alike.
- **Scan coverage summary: files scanned + language breakdown.** The engine gains a `-summary <path>` flag (mirroring `-agility`) that writes a scan-summary JSON ŌĆö `files_scanned`, `files_skipped`, `findings`, and `files_by_language` (counted per detected language by the walker). The app persists it on the scan run (`FilesScanned`, `LanguagesJson`; additive bootstrap migration), exposes it on the scan DTO (`filesScanned`, `languages`), and the project page shows a fourth **Files scanned** card: an SVG donut chart of the per-language distribution (total in the center, color-coded legend, languages beyond the 8-color palette grouped as "other").
- **Delete project.** `DELETE /api/v1/projects/{id}` removes a project together with everything derived from it ŌĆö findings, scan runs, and (when no other project references the same archive) the uploaded `.zip`. Refused with `409 scan_in_progress` while a scan is running. The web app's project page gains a two-step **Delete project** button (confirm/cancel) that returns to the projects list on success.

### Fixed
- **JS/TS AST walker crash on `try`/`catch` without `finally`.** goja's `TryStatement.Finally` is a nil `*BlockStatement` when the clause is absent; passed through `walk`'s interface parameter it became a typed nil that the nil check missed, and the scanner segfaulted ŌĆö killing the whole scan ŌĆö on any JavaScript/TypeScript file containing a bare `try`/`catch`. The walker now guards typed-nil nodes (reflection check + explicit `Finally` guard), pinned by regression tests.
- **Scans are now panic-proof per file.** A detector panic on one pathological file is recovered, logged, and counted as a skipped file (`detector_panic: ŌĆ”`) instead of aborting the scan ŌĆö same fail-soft policy the scanner already applied to per-file read/parse errors.

## [0.2.0] - 2026-06-11

### Added
- **Hand-rolled crypto: constant-fingerprint pack.** A new `rules-community/fingerprints` pack (language `any` ŌĆö runs on every scanned file) detects FindCrypt-style magic constants that essentially only occur in a crypto *implementation*: the RSA public exponent (65537 / 0x10001), the AES forward/inverse S-boxes, the SHA-256 K table and IV, the SHA-1/MD5 IV pair and MD5 sine table, the RFC 3526 MODP prime, the Curve25519 prime / a24 constant, and the secp256k1 field prime ŌĆö plus a Go `big.Int` modexp heuristic (`GO_BIGINT_MODEXP`). Ambiguous single constants are low severity/confidence by design (promotion fuel ŌĆö see below); unmistakable ones (the MODP prime, the secp256k1 tail) carry high severity on their own. Rule count 714 ŌåÆ 725 across 47 packs.
- **Multi-signal promotion of hand-rolled crypto (`scanner/promote.go`).** When Ōēź2 *distinct* hand-rolled / fingerprint rules agree on the same algorithm within one file (with an RSA/DH fold for the modexp family), the scanner appends one `HANDROLLED_<ALG>_PROMOTED` finding at **high** severity with Bayesian-fused confidence (1 ŌłÆ ╬Ā(1 ŌłÆ cßĄó), capped at 0.95 ŌĆö the same form as package fusion, deliberately kept within-file). Corroborated hand-rolled crypto now surfaces as a high-confidence finding instead of a scatter of 0.35ŌĆō0.4 heuristic leads.
- **Unmapped-crypto coverage sentinel (`scanner/sentinel.go`).** If a file imports a known *classical* crypto library (Python `cryptography` / PyCryptodome / M2Crypto / paramiko / jwtŌĆ”, JS/TS `node:crypto` / jsonwebtoken / node-forge / ellipticŌĆ”, `java.security` / `javax.crypto` / BouncyCastle, Go `crypto/*` / `x/crypto`, .NET `System.Security.Cryptography`, Ruby/PHP/Rust entries) but the entire detection stack recognized **zero** crypto findings in it, the scanner emits one informational `CRYPTO_API_UNMAPPED` finding naming the library (severity `info`, `quantum_safety: unknown`, category `coverage-sentinel`) ŌĆö possible coverage gaps become visible findings instead of silence. PQC libraries (oqs, circl, Kyber, Dilithium, ML-KEM, ML-DSA, SPHINCS+ŌĆ”) are excluded and never trigger it. The validation corpus gains a matching informational bucket **S**; the gate is now 51 instances (50 matched + 1 designed AST-tier skip), 60/60 findings precision-clean.
- **Per-rule inline examples, ratchet-enforced.** The rule schema gains `examples: {match: [ŌĆ”], no_match: [ŌĆ”]}` ŌĆö inline self-tests where every `match` line must trigger the rule and every `no_match` line (a near-miss boundary, encoding today's bug regressions) must not, validated by `TestRuleExamples`. `TestRuleExamplesRatchet` + `packages/go/rules/examples_baseline.txt` (a frozen grandfather list of 502 ids) make examples **mandatory for new regex rules** and shrink-only when baselined rules are backfilled; 163 regex rules carry examples today (152 backfilled across python / go / java / javascript / typescript / csharp / nginx / ssh / config / openssl / apache, plus the new packs).
- **X.509 certificate scanning.** Standalone certificate / key files (`.pem`, `.crt`, `.cer`, `.der`, `.key`) are now routed (language `x509`) to a dedicated detector (`packages/go/detectors/x509`) that parses PEM blocks and raw DER. Certificates emit **two** findings ŌĆö the public-key algorithm and the signature algorithm (`X509_CERT_PUBKEY_*` / `X509_CERT_SIG_*`); CSRs, private keys (`X509_PRIVATE_KEY_*`, CWE-798), and public keys are flagged too. Classical-crypto matches are critical; snippets contain only the `-----BEGIN ŌĆ”-----` marker line ŌĆö never key material ŌĆö and unknown / PQC OIDs are never flagged vulnerable. Hooked into the normal scan pipeline, so JSON/JSONL/SARIF and the CLI work unchanged.
- **SSH config scanning.** `sshd_config` / `ssh_config` are now routed (language `ssh-config`) to a new `rules-community/ssh` rule pack covering `HostKey`, `KexAlgorithms`, `Ciphers`, `HostKeyAlgorithms`, and `PubkeyAccepted*` directives.
- **Validation gate.** A labeled ground-truth corpus (`fixtures/validation-corpus/` ŌĆö Python / Go / JS / Java / C# / nginx / sshd_config / X.509 certs / vendored code / dependency manifests; 40 graded instances, 10 forbidden no-false-positive locations incl. PQC code, and deps expectations in `expected-findings.yaml`) gated by `packages/go/validationgate` (`go test ./validationgate/...`). The gate runs the real scan + deps pipeline and enforces recall, zero false positives on PQC code, **strict precision** (every emitted finding must map to ground truth), and the deps expectations. Green on the regex-floor build: 39/40 matched (1 AST-tier skip), 45/45 findings precision-clean, deps 7/7 + 4 must-not-flag clean.
- **63 new rules (651 ŌåÆ 714)** closing quantum-broken detection families: Python DSA / DH / Ed25519 / X25519 / ECDH, aliased-import rules, a hand-rolled-RSA heuristic (low confidence), and 3DES; Go `ed25519` / `ecdh` / `curve25519` and classical `tls.TLS_*` cipher-suite constants; Java `Signature` (RSA/ECDSA/DSA), `KeyAgreement` (DH/ECDH), DH keygen, and EdDSA; JS/TS JWT `RS*`/`PS*`/`ES*` algorithms (sign + verify allowlists), `createECDH`, `createSign` RSA, and aes-128 / 3DES `createCipheriv`; C# 3DES + ECDH providers; nginx `ssl_ciphers` classical key exchange (`NGINX_SSL_CIPHERS_CLASSICAL_KX`). Core quantum-broken rules also now ship a one-line PQC `recommendation` in the OSS pack (previously detection-only).
- **GitHub Action + CI scanner image.** A Docker-based GitHub Action (`github-action/`) runs `relixq scan` (`scan-type: code | deps | tls`), writes SARIF, and exposes its path as an output for `github/codeql-action/upload-sarif` ŌĆö so PR Code Scanning annotations and build-gating (`fail-on`) work out of the box. Backed by a slim scanner image (`Dockerfile.scanner`, CGO-off: regex floor + pure-Go Go/JS/TS AST) published to GHCR by `.github/workflows/relixq-image.yml`. Example workflow in `docs/ci-examples/github.yml`.
- **TLS / certificate scanning (`relixq scan tls`).** Probes TLS endpoints (positional `host[:port]` args, repeated `--target`, or a `--targets` file) and flags quantum-vulnerable and weak transport crypto: classical certificate keys (RSA/ECDSA/DSA), undersized RSA keys, SHA-1 signatures, expired/expiring certs, self-signed leaves, deprecated TLS 1.0/1.1, and weak negotiated cipher suites. Self-contained (`packages/go/tlsscanner`, stdlib `crypto/tls` only ŌĆö no Service Bus, no fleet machinery) and shares the `--format` / `--severity-threshold` / `--baseline` pipeline, so it emits SARIF for CI.
- **Dependency scanning (`relixq scan deps`).** Walks a project's dependency manifests ŌĆö requirements.txt, Pipfile, pyproject.toml, package.json, go.mod ŌĆö and flags declared packages known to implement quantum-vulnerable cryptography, using an embedded knowledge base (no network or lockfile resolution). Runs in-process (no scanner subprocess) and shares the `--format` / `--severity-threshold` / `--baseline` pipeline with `relixq scan`, so `relixq scan deps --format sarif` uploads to CI.
- **SARIF for GitHub Code Scanning.** `relixq scan --format sarif` now emits SARIF 2.1.0 with `security-severity`, classification `tags` (incl. CWE taxa and `pqc`), per-rule `help.markdown`, and stable `partialFingerprints` ŌĆö uploadable directly to GitHub Code Scanning. `--format json` exports the full structured findings.
- **Baselines.** `relixq baseline` records current findings to `.relixq-baseline.json`; `relixq scan --baseline <file>` reports only findings absent from it, so CI can gate on *new* quantum-vulnerable crypto. Matching is by a content fingerprint (rule + file + snippet) resilient to line-number drift. (Path/line suppression via `.relixqignore` and inline `// relixq-ignore` was already supported.)
- **Demo image ships full AST.** `apps/api/Dockerfile` now builds the scanner engine with CGO (Debian builder) so the Tree-sitter runners compile ŌĆö C/C++, Java, Ruby, Rust, Swift, Julia, Scala, Kotlin ŌĆö and bundles the `relixq-roslyn` C# subprocess. `docker compose up` therefore gets full AST precision across every supported language with no toolchain on the host. A plain `go build` (no CGO) still works and degrades to the regex floor.
- **C# AST detection (Roslyn).** The `relixq-roslyn` subprocess is included in the OSS repo (`packages/go/detectors/csharpast/tools/relixq-roslyn`, Apache-2.0) and the C# AST runner is wired into the engine. When the binary is present, C# crypto-api primitives (RSA/ECDSA/ECDH) are matched at the syntax level (anchored to AST nodes, not text) and AST-only rules like `CipherMode.ECB` fire; when it is absent the scanner falls back to the regex floor with a one-line notice ŌĆö never an error. Regex and AST rules share ids; the scanner dedups by `(rule id, line)`, preferring the higher-fidelity AST finding, so baselines/fingerprints stay stable regardless of whether Roslyn is available. (Tree-sitter languages remain regex-only pending a CGO build stage.)
- **PQC detection (two-layer model).** All 192 `crypto-api` rules now ship a **detection layer** in `rules-community/` (across 33 languages ŌĆö RSA, ECDSA, ECDH, DSA, Ed25519, and more) so the OSS scanner sees the quantum-vulnerable primitives it is named for. The community rule pack grows from ~455 to ~647 rules. AST-based detections are carried as a regex recall floor (the OSS engine ships no AST runner yet); precision is restored later by activating the AST runners. The migration **enrichment layer** (NIST/FIPS substitutions ŌĆö e.g. ML-KEM/FIPS 203, ML-DSA/FIPS 204 ŌĆö and references) is an optional overlay: set `-rulepack <dir>` / `RELIXQ_RULE_PACK` and the engine merges it onto matching findings in place via `packages/go/enrich`. The OSS build ships detection only; without an overlay, findings are detection-complete.
- Project source kinds beyond the bundled sample: scan a **git repository** (with an optional access token for private repos), a **local path** under a read-only mounted folder (`LOCAL_SCAN_PATH` ŌåÆ `/scan`), or an **uploaded `.zip`** archive (extracted server-side with zip-slip / zip-bomb guards and kept in a Docker volume for re-scans; up to 1 GB). Web API helpers fail soft on transport errors so an oversized upload shows a message instead of a server-error page.
- Self-contained OSS demo app: `docker compose up --build` brings up Postgres + the .NET API + the Next.js web UI on a single network with no external dependencies.
- `apps/api` ŌĆö ASP.NET minimal API (.NET 8) with local email/password auth, projects, scan runs, and risk scoring; shells the bundled `relixq-scan-code` engine and parses JSONL findings.
- `apps/web` ŌĆö Next.js web app: signup/login, project creation, scan trigger, and a results view (risk-score gauge, crypto-agility grade, findings table).
- OSS libraries: the app consumes `RelixQ.Contracts`, `RelixQ.Scoring`, `RelixQ.Auth.Local`, and `@relix-q/web-components`.
  `RelixQ.AI.BYOK` and `@relix-q/web-client` are standalone reusable packages.
- Bundled intentionally-vulnerable sample (`fixtures/sample-vulnerable/`) for a deterministic, offline demo scan.
- BYOK LLM adapter package prep via `RelixQ.AI.BYOK`; the OSS app does not wire AI explanations into its endpoints yet.

### Changed
- **Three-tier `quantum_safety` taxonomy.** The risk-tagged values are now `vulnerable` (Shor-broken asymmetric ŌĆö unchanged meaning, backward compatible), new `grover_weakened` (AES-128 / 3DES / Ōēż128-bit symmetric and quantum-inventory hash rules ŌĆö 40 rules), and new `classically_broken` (MD5 / SHA-1 / RC4 / DES / RC2 / MD4 / RIPEMD ŌĆö 195 rules), so classically-broken findings are no longer conflated with the quantum inventory. Consumers updated end-to-end: SARIF classification tags, the deps knowledge base, the TLS scanner's weak-cipher table, `RelixQ.Scoring` enums + `ScoringFormulaV1` (Grover-weakened caps base risk at 6; classically-broken floors it at 7), the `apps/api` scoring service, the `RelixQ.Contracts` JSON schema, the `@relix-q/web-client` zod schema, and `@relix-q/web-components` types + rule-browser badges.
- **Severity parity inside the quantum-broken bucket.** 30 elliptic-curve / finite-field rules (ECDSA, ECDH, EdDSA, DH, DSA) raised from high to critical to match RSA ŌĆö both fall to Shor's algorithm.

### Fixed
- **`IAC_DOCKERFILE_OLD_TLS` dead regex.** The pattern opened with `\b` before `--`, which can never assert between whitespace and a hyphen ŌĆö the rule was unmatchable and had never fired. Once that was removed, the bare `--tlsv1` alternative would have prefix-matched the compliant `--tlsv1.2` / `--tlsv1.3`; both bugs are fixed and pinned with boundary `no_match` examples. Caught by the new per-rule examples machinery during backfill.
- **`JS_NODE_CRYPTO_RSA` wrong-algorithm dispatch.** `generateKeyPair` / `generateKeyPairSync` findings now dispatch on the first argument instead of always reporting RSA; new `JS_NODE_CRYPTO_EC` / `ED25519` / `DH` / `X25519` rules (with TypeScript twins) cover the non-RSA curves.
- **nginx `ssl_protocols` false positive.** `NGINX_SSL_PROTOCOLS_DEPRECATED` no longer flags the compliant `ssl_protocols TLSv1.2 TLSv1.3;` (the regex matched the `TLSv1` prefix inside `TLSv1.2`); now token-anchored. Same fix for `NGINX_PROXY_SSL_PROTOCOLS_DEPRECATED`.
- **`JAVA_DES_CIPHER` 3DES conflation.** Split into single-DES (high) and a new `JAVA_3DES_CIPHER` (3DES/DESede, medium) so 3DES is reported with the correct algorithm and severity.
- **Directory-only ignore patterns.** `.relixqignore` patterns like `node_modules/` never matched the files beneath the directory; suppression now follows gitignore semantics.

## [0.1.0] - 2026-05-18

### Added
- Initial public release.
- Scanner engine: walker, language router, rule loader, AST + regex detectors.
- Pure-Go libraries: `scanner`, `finding`, `rules`, `detectors`, `suppression`, `diffmode`, `blame`.
- IP-claim packages: `agility` (Crypto-Agility Scorecard), `fusion` (Bayesian cross-vertical fusion), `graph` (blast-radius graph), `sbom` (SBOM ingest), `migrationplan` (remediation planning).
- `relixq` CLI and `relixq-scan-code` standalone scanner.
- AST detectors for 13 languages (Go, C#, Java, JS/TS, Python, Rust, C/C++, Kotlin, Swift, PHP, Ruby, Scala, Julia); subprocess-backed runners for C#, Python, and PHP.
- Community rule packs for 45+ languages and config formats, covering MD5, SHA-1, DES, 3DES, RC4, RSA-1024, TLS 1.0/1.1, and hardcoded keys.

# Getting support

Thanks for using Relix-Q OSS. Picking the right channel helps maintainers answer
quickly and keeps private information private.

## Where to go

| You want to... | Go to |
|---|---|
| Ask a setup or usage question | GitHub Discussions Q&A |
| Propose an idea or roadmap change | GitHub Discussions Ideas |
| Report a reproducible bug | Open a bug issue |
| Request a scanner rule or language improvement | Open a feature issue |
| Report a security vulnerability privately | See [SECURITY.md](SECURITY.md) |
| Contribute code or docs | See [CONTRIBUTING.md](CONTRIBUTING.md) |
| Understand project decisions | See [GOVERNANCE.md](GOVERNANCE.md) |

If Discussions are not enabled yet, open an issue and label it as a question.

## Before opening an issue

1. Search existing issues and discussions.
2. Check [README.md](README.md) for quickstart, scan modes, CI output, and
   suppression/baseline usage.
3. Check [docs/CONFIGURATION.md](docs/CONFIGURATION.md) and
   [docs/DEPLOYMENT.md](docs/DEPLOYMENT.md) for environment and compose setup.
4. Check [docs/DEVELOPMENT.md](docs/DEVELOPMENT.md) for local build commands.
5. Check [docs/TROUBLESHOOTING.md](docs/TROUBLESHOOTING.md) for common clone,
   Docker, scan, and build problems.
6. For private security reports, do not paste details into a public issue.

## What support means

Relix-Q OSS is community-supported. Maintainers respond on a best-effort basis.
There is no SLA for issues or pull requests.

## Help us help you

- Include the Relix-Q version, commit SHA, OS, Docker version, Go version, .NET
  SDK version, and Node/npm versions when relevant.
- Include the exact command and error output.
- Use the bundled sample or a tiny sanitized repro when possible.
- Redact repository URLs, private tokens, private source paths, and key material.
